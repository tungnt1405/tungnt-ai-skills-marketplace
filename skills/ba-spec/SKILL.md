# ba-spec

Use `ba-spec` to turn business input into BA feature specifications for dev/QA handoff in the conversation language unless the user explicitly asks otherwise. Do not lock output to one human language.

This file is the router and execution contract. Keep it short. Load the referenced rule files only when their trigger applies.

## Manual Activation Only

Use this skill only when the user explicitly invokes `ba-spec`.

Do not run it at install time. Do not auto-process during session bootstrap.

Manual utility only. Not a process skill. Not a domain lens.

## Non-negotiable output rule

All user-facing outputs generated by this skill must follow the conversation language unless the user explicitly asks otherwise:

- Markdown feature spec.
- HTML feature spec.
- Clarification questions.
- Dev/QA checklist.
- Evidence log.
- Assumptions and open questions.

Keep machine-readable IDs and source tags in English:

- IDs such as `FR-001`, `BR-001`, `AC-001`, `Q-001`.
- Source tags: `[PROVIDED]`, `[FIGMA]`, `[FILE]`, `[INFERRED]`, `[ASSUMPTION]`, `[OPEN_QUESTION]`.
- Gherkin keywords may remain `Given / When / Then / And` unless the user asks for localized BDD style.

If the user explicitly asks for Vietnamese-only output, obey that request.
If the user explicitly asks for a bilingual or English-only output, obey that request.

For details, read `references/output-language-rules.md`.

## When to use

Use this skill when the user asks to:

- Create a feature specification.
- Convert business requirements into dev-ready documentation.
- Produce a handoff document for dev/QA.
- Describe a new feature.
- Describe a feature upgrade or change request.
- Use business text, Figma links, Figma screenshots, old specs, tickets, PDFs, spreadsheets, meeting notes, API notes, or other related files as input.
- Generate both Markdown and HTML documentation.

## BA-only Figma boundary

If the request is only for BA/PO/spec output, do not invoke any UI coding workflow. Keep `ba-spec` limited to business analysis and handoff artifacts.

If the input is Figma-related and developer handoff needs UI implementation guidance, consult `figma-to-code` as supporting reference for implementation details. Do not turn that into automatic UI coding for BA-only requests.

## When not to use

Do not use this skill to:

- Implement production code.
- Invent API, database, architecture, queue, security, or infrastructure design.
- Treat Figma UI as confirmed business rules.
- Replace PO/stakeholder approval.
- Hide uncertainty.

## Mandatory workflow

Follow this exact workflow.

1. **Classify input** using `references/input-classification.md`.
2. **Resolve output package path** using `references/output-packaging-rules.md` before creating any final deliverable.
3. **Apply workspace hygiene rules** using `references/workspace-hygiene-rules.md` before creating temporary scripts or helper files.
4. **If any `figma.com` URL or Figma screenshot exists, run the Figma evidence gate** using `references/figma-mcp-gate.md` before writing or editing the spec.
5. **Preserve every original Figma URL** in spec output; dev-facing output must include clickable Figma links, not only node IDs.
6. **Extract evidence and confidence tags** using `references/source-confidence-and-evidence.md`.
7. **Apply BA practice rules** using `references/ba-documentation-principles.md`.
8. **If this is a feature upgrade**, apply `references/feature-upgrade-rules.md`.
9. **If related files are provided**, apply `references/file-handling-rules.md`. **If a file embeds screenshots/diagrams** (e.g. `.docx`/`.pptx`/`.pdf`), extract them using `references/file-image-extraction-rules.md`, copy into the package `assets/`, and tag `[FILE]`.
10. **Generate the Markdown spec in the output language** inside the output package using `templates/feature-spec.md` and `references/spec-generation-workflow.md`.
11. **Generate the HTML spec in the output language** inside the same output package using `templates/feature-spec.html` and `references/html-rendering-rules.md`. If extracted screenshots exist, use the step-by-step document variant (image beside each step + CSS lightbox). If the Markdown contains Mermaid diagrams, the HTML must render them using Mermaid.js CDN and also keep a readable source fallback.
12. **Delete temporary helper files** using `references/workspace-hygiene-rules.md`.
13. **Run quality gates** using `references/quality-gates.md` before final response.

## Figma hard gate

If the input contains at least one Figma URL, you must not directly generate or edit `feature-spec.md` or `feature-spec.html` until one of these is true:

- You successfully used a Figma MCP tool to inspect the linked file/frame/node and recorded extracted evidence; or
- You attempted Figma MCP access, it failed or was unavailable, and you recorded a Figma evidence log in the output language explaining the failure and fallback status; or
- The user explicitly says to ignore Figma/MCP and proceed from business text only.

A generic sentence such as “I used the Figma links” is not enough. The spec must include evidence rows per Figma link or an explicit failure row per link.

When Figma MCP fails or is unavailable, do not pretend that Figma was analyzed. Mark Figma-derived data as `[OPEN_QUESTION]` or ask for fallback screenshots/exported frames/copied text.

Detailed rules: `references/figma-mcp-gate.md`.

## Source confidence tags

Use the exact tags below:

- `[PROVIDED]`: directly stated by the user.
- `[FIGMA]`: extracted from Figma screenshot/link/MCP.
- `[FILE]`: extracted from related file.
- `[INFERRED]`: logically inferred from supplied evidence.
- `[ASSUMPTION]`: assumed only to make a draft usable; requires confirmation.
- `[OPEN_QUESTION]`: unresolved question affecting scope, behavior, testability, or implementation.

If unsure, use `[OPEN_QUESTION]`.

## Required output package

When the user asks for deliverables, create one final package directory, not loose files in the project root.

Default package path:

```text
docs/tungnt-ai-skills/ba-spec-output/{{YYYYMMDD}}/{{feature-slug}}/
```

Do not embed full Epic and Story names in the folder path by default. Keep Epic/Story in the metadata sections inside `feature-spec.md` and `feature-spec.html`. If the user explicitly asks to group by Epic, use the optional grouped format documented in `references/output-packaging-rules.md`.

### Default final deliverables

By default, create **only these two user-facing files** inside the package:

```text
feature-spec.md
feature-spec.html
```

Do not create separate `README.md`, `figma-links.md`, `evidence/`, `handoff/`, `questions/`, or `change-impact-summary.md` files unless the user explicitly asks for extra companion files, debug artifacts, or separated handoff indexes.

All required handoff information must be embedded in the two default deliverables:

- Figma original links and MCP evidence log: embed in `feature-spec.md` and mirror in `feature-spec.html`.
- Dev/QA checklist: embed as a section in `feature-spec.md` and mirror in `feature-spec.html`.
- Clarification questions: embed in the output-language open-questions section in `feature-spec.md` and mirror in `feature-spec.html`.
- Epic, Story, Feature, package path, date, status, and source metadata: embed in `feature-spec.md` and mirror in `feature-spec.html`.

Do not leave `feature-spec.md`, `feature-spec.html`, `extract_texts.py`, `generate_html.py`, `output.txt`, `extracted_texts.txt`, or similar helper files in the project root. If such files are accidentally created, move final deliverables into the package and delete helper files before final response.

Detailed rules: `references/output-packaging-rules.md` and `references/workspace-hygiene-rules.md`.

## Requirement ID conventions

Use these prefixes:

- `BG-###`: Business Goal
- `STK-###`: Stakeholder
- `ROLE-###`: Role / Actor
- `US-###`: User Story
- `FLOW-###`: Flow Step
- `FR-###`: Functional Requirement
- `BR-###`: Business Rule
- `DR-###`: Data Requirement
- `VR-###`: Validation Rule
- `PERM-###`: Permission Rule
- `STATE-###`: State
- `TR-###`: State Transition
- `EC-###`: Edge Case
- `ERR-###`: Error Handling
- `AC-###`: Acceptance Criteria
- `NFR-###`: Non-Functional Requirement
- `AUD-###`: Audit / Analytics / Logging
- `DEP-###`: Dependency
- `CHG-###`: Change Impact
- `ASM-###`: Assumption
- `Q-###`: Open Question
- `UIQ-###`: UI Open Question (unknown select catalog or unclear button/element — ask, never guess)

## Final response contract

After generating files, respond in the output language with:

1. Final output package path.
2. Files created/updated inside the package.
3. Whether temporary helper files were cleaned up.
4. Whether Figma MCP was used successfully.
5. Count of Figma links inspected, failed, or skipped.
6. Confirm that original clickable Figma links are embedded in both `feature-spec.md` and `feature-spec.html`.
7. Blocking open questions.
8. Whether the spec passed quality gates.

Do not claim Figma was analyzed unless the evidence log proves it. If diagrams are present, confirm that the HTML includes Mermaid.js rendering support.
